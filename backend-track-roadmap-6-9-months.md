# Backend Developer Roadmap: 9 месяцев с нуля

> Версия плана: реалистичный основной маршрут для ученика без опыта. Нагрузка: до 20 часов в неделю, около 80 часов в месяц и 720 часов за девять месяцев.

## Честная рамка результата

Этот трек не обещает трудоустройство за фиксированный срок. За 9 месяцев при регулярной практике ученик может выйти на уровень, с которым разумно начинать отклики на позиции `Backend Intern`, `Junior Python Developer` или `Junior Backend Developer`: понимать свой код, собирать и защищать API-проект, работать с базой данных, тестами, Docker и GitHub.

**Шесть месяцев** здесь не являются сокращённой версией готового backend-разработчика. Это контрольная точка: ученик способен сделать небольшой API с PostgreSQL, базовой авторизацией и тестами, но ему ещё нужны опыт проектирования, повторение и завершённый портфельный проект.

**Девять месяцев** — рекомендуемая длительность для человека с нуля. Три последних месяца нужны не для добавления модных технологий, а для нормального закрепления: качества кода, сложных связей, командного процесса, самостоятельной работы и защиты проекта.

## Для кого рассчитан трек

- Ученик начинает без опыта программирования или с фрагментарными знаниями Python.
- Есть компьютер, стабильный доступ к интернету и возможность регулярно писать код.
- Ученик занимается пять дней в неделю, а не пытается компенсировать пропуски одним длинным днём.
- Наставник помогает с обратной связью и разбором ошибок, но не пишет решение вместо ученика.

## Стек курса

Основной стек: `Python`, виртуальные окружения и зависимости, `Git` и GitHub, `HTTP`, `FastAPI`, `PostgreSQL`, SQL и `SQLAlchemy`, `Alembic`, `pytest`, линтер и formatter, `Docker`, базовый Linux, Redis, простые фоновые задачи, CI и деплой.

Осознанно не включаем в обязательную часть: микросервисы, Kubernetes, Kafka/RabbitMQ в глубину, DDD как теоретическую систему, сложную асинхронность, Clean Architecture как самоцель, облачную инфраструктуру и глубокую оптимизацию PostgreSQL. Ученик должен сначала научиться уверенно делать один понятный монолитный сервис.

## Как устроен месяц

Каждый месяц содержит **20 обязательных платформенных занятий по 40–60 минут**:

- 12 основных уроков: новая тема, разбор кода, небольшой управляемый пример;
- 4 практикума: самостоятельная работа по сценарию, но с чёткими критериями;
- 2 проектные сессии: интеграция изученного в сквозной проект;
- 1 занятие на повторение и диагностику пробелов;
- 1 защита или разбор с наставником.

Одних двадцати уроков недостаточно для нагрузки 20 часов в неделю. После каждого блока ученик выполняет кодовые задания, читает ошибки, исправляет проект и оформляет результат. Реалистичное распределение 80 часов в месяц:

| Вид работы | Часы в месяц | Смысл |
| --- | ---: | --- |
| Платформенные занятия | 16–20 | Теория, короткая практика, контроль понимания |
| Обязательные упражнения | 22–26 | Повторение приёмов до уверенного применения |
| Сквозной проект | 22–26 | Код, отладка, README, рефакторинг |
| Повторение и разбор ошибок | 8–10 | Возврат к слабым местам, а не движение «по верхам» |
| Наставник и защита | 4–6 | Код-ревью, вопросы, план исправлений |

### Ритм одной недели

| День | Формат                                                       |
| ---- | ------------------------------------------------------------ |
| 1    | Новый урок, короткое упражнение, конспект решений            |
| 2    | Новый урок, 2–4 небольшие задачи                             |
| 3    | Практикум: отдельная функциональность проекта                |
| 4    | Новый урок, рефакторинг вчерашнего кода                      |
| 5    | Повторение, тесты или отладка, Git-коммит, вопрос наставнику |

Четвёртая неделя месяца не должна быть полностью занята новой теорией. Её задача — собрать проект, устранить ошибки, пройти проверку и объяснить решения своими словами.

## Два темпа обучения

### Основной маршрут

Обязательная часть рассчитана на 20 часов в неделю. Переход к следующему месяцу возможен только после контрольной точки. Если работа не принята, ученик использует время недели закрепления для исправления, а не начинает новые сложные темы.

### Маршрут для активных учеников

В каждом месяце предусмотрено **4–8 дополнительных занятий по 40–60 минут** и отдельные усложнённые практики. Они не являются «домашней работой для всех» и не должны создавать ощущение отставания у основного потока.

Дополнительные занятия строятся по трём правилам:

- опираются только на уже пройденную обязательную базу;
- дают отдельный измеримый артефакт: тест, скрипт, улучшение API, техническое решение;
- не заменяют повторение и работу над основным проектом.

Активный ученик может тратить на трек 24–28 часов в неделю. Его цель — не раньше «закрыть программу», а получить больше самостоятельных решений и качественный GitHub-профиль.

## Сквозной проект

На протяжении трека ученик развивает один домен: **StudyHub — сервис для учёта учебных задач и обучения**. Домен растёт вместе с навыками, поэтому каждая новая технология отвечает на уже понятную проблему.

| Этап    | Версия StudyHub          | Что появляется                                        |
| ------- | ------------------------ | ----------------------------------------------------- |
| Месяц 1 | Console Planner          | Задачи в памяти программы, меню, проверки ввода       |
| Месяц 2 | Persistent Planner       | JSON, модули, классы, сохранение и загрузка           |
| Месяц 3 | Planner API              | REST API и документация без базы данных               |
| Месяц 4 | Planner API + PostgreSQL | Таблицы, SQL, сохранение данных                       |
| Месяц 5 | Structured API           | ORM, миграции, связи и конфигурация                   |
| Месяц 6 | Personal StudyHub        | Пользователи, авторизация, права и тесты              |
| Месяц 7 | Deployable StudyHub      | Docker, логирование, CI, первый деплой                |
| Месяц 8 | StudyHub LMS Core        | Курсы, модули, уроки, записи, прогресс                |
| Месяц 9 | StudyHub Release         | Кеш, фоновые операции, финальное качество и портфолио |

# Месяц 1. Компьютерная база, Python и алгоритмическое мышление

## Цель

Снять технический барьер: ученик умеет открыть терминал, создать и запустить Python-файл, прочитать ошибку, сохранить изменения в Git и написать небольшую консольную программу. Backend-термины на этом этапе даются только как мотивация, а не как предмет изучения.

## Обязательные уроки

1. Как компьютер выполняет программу: файл, интерпретатор, инструкция, ошибка.
2. Папки, пути, расширения, терминал и безопасная навигация по файловой системе.
3. Установка Python, редактор кода, запуск скрипта, виртуальное окружение, `pip` и проверка версии интерпретатора.
4. Git: репозиторий, статус, коммит, история; зачем разработчику история изменений.
5. Переменные, типы `int`, `float`, `str`, `bool` и присваивание.
6. Ввод, вывод, преобразование типов и f-строки.
7. Строки: индексы, срезы, методы, типичные ошибки с пробелами и регистрами.
8. Сравнения, логические операторы и осмысленные условия.
9. Ветвления `if`, `elif`, `else`: как формулировать правило до написания кода.
10. Циклы `for` и `while`, счётчики, `break`, `continue`, защита от бесконечного цикла.
11. Списки, кортежи, множества и словари: выбор структуры данных для простой задачи.
12. Первые функции, декомпозиция и чтение traceback; отладка через маленькие шаги.

## Практикумы и проектные сессии

- Практикум 1: 12 коротких задач на преобразование типов, строки и условия.
- Практикум 2: задачи на циклы, поиск, подсчёт и валидацию пользовательского ввода.
- Практикум 3: список задач и словарь статистики без файлового хранения.
- Практикум 4: Git-практика: несколько осмысленных коммитов, `.gitignore`, первый README.
- Проектная сессия 1: меню и основные команды `Console Planner`.
- Проектная сессия 2: обработка некорректного ввода, чистка повторяющегося кода, демонстрация проекта.

## Обязательный результат

`Console Planner` в памяти программы: добавить задачу, вывести список, отметить выполненной, удалить, отфильтровать по статусу. Данные после перезапуска могут исчезать: хранение в файлах ещё не изучалось.

## Контрольная точка

Ученик самостоятельно создаёт репозиторий, запускает программу из терминала, объясняет работу цикла и функции, исправляет ошибку по traceback, делает коммит и показывает README.

## Дополнительно для активных

- Множества, кортежи и случаи, где они удобнее списка.
- Простая сортировка и параметр `key`.
- Генераторы списков без сложных вложенных конструкций.
- Валидация даты и приоритета задачи через отдельные функции.
- История выполненных задач в памяти и простая статистика.
- 10 задач на чтение чужого кода: предсказать результат до запуска.

## Не включать раньше времени

Файлы, JSON, ООП, API, базы данных и асинхронность. Попытка сделать «настоящий backend» здесь обычно маскирует неусвоенные основы Python.

# Месяц 2. Python глубже, файлы и структура небольшого проекта

## Цель

Научить ученика превращать программу из одного файла в понятный небольшой проект: выделять функции, модули и модели данных; сохранять данные; не скрывать ошибки; писать первые тесты.

## Обязательные уроки

1. Функции как контракт: параметры, `return`, побочные эффекты, проверка результата.
2. Значения по умолчанию, область видимости, изменяемые объекты и частая ошибка с `[]` по умолчанию.
3. `*args`, `**kwargs` и распаковка: подготовка к чтению реального Python-кода.
4. Исключения и отладка: `try`, `except`, `else`, `finally`, что нельзя перехватывать широко.
5. Модули, импорты, точка входа, пакеты и `if __name__ == "__main__"`.
6. Структура небольшого проекта: `main`, `models`, `services`, `storage`, `validators`; зависимости проекта и `requirements.txt`.
7. Файлы, `pathlib` и контекстный менеджер `with`: кодировки, безопасное чтение и запись текста.
8. JSON и CSV: сериализация, ошибки данных, резервная копия и восстановление.
9. Comprehensions, итераторы и генераторы: ленивые вычисления без преждевременной сложности.
10. Классы и объекты: когда модель полезнее словаря, а когда класс не нужен.
11. `self`, атрибуты, методы, `__str__`, класс-атрибут и объект-атрибут.
12. Инкапсуляция, свойства, композиция, наследование без злоупотребления и `dataclass`.
13. Декораторы: синтаксис `@`, функции высшего порядка, замыкание и практические примеры `@property` и `@pytest.fixture`.
14. Type hints, `pytest`, фикстуры и параметризация; первый статический анализ кода через выбранный линтер.

## Практикумы и проектные сессии

- Практикум 1: разбить монолитную программу на функции с ясными названиями.
- Практикум 2: безопасно прочитать и записать JSON; обработать пустой и повреждённый файл.
- Практикум 3: описать задачу через `dataclass`, добавить валидацию и строковое представление.
- Практикум 4: написать 5–8 тестов для чистых функций и слоя хранения.
- Проектная сессия 1: перенести `Console Planner` в модули и сделать хранение в JSON.
- Проектная сессия 2: README с установкой, запуском, известными ограничениями и примерами команд.

## Обязательный результат

`Persistent Planner`: консольный проект с хранением задач в JSON, отдельными модулями, обработкой ожидаемых ошибок и минимум пятью тестами.

## Контрольная точка

Ученик объясняет разницу между функцией и методом, между списком и объектом, понимает путь к файлу, не использует голый `except`, запускает тесты и может показать, где находится точка входа проекта.

## Дополнительно для активных

- `Enum` для статусов и приоритетов.
- Сортировка, поиск и фильтрация через отдельный слой сервиса.
- Экспорт задач в CSV и создание резервных копий JSON.
- `logging` на базовом уровне вместо отладочных `print`.
- Параметризация тестов `pytest.mark.parametrize`.
- Анализ и исправление специально подготовленного «грязного» Python-файла.

## Не включать раньше времени

Паттерны проектирования, сложные абстрактные классы, ORM и многослойную архитектуру. Важнее научиться видеть ответственность каждого файла.

# Месяц 3. Интернет, HTTP и первый REST API

## Цель

Показать, как клиент получает данные от backend: запрос, ответ, JSON, HTTP-метод, статус и документация. Ученик создаёт API в FastAPI, но пока не подключает базу данных.

## Обязательные уроки

1. Интернет на прикладном уровне: клиент, сервер, адрес, порт, DNS, localhost.
2. HTTP-запрос и HTTP-ответ: URL, заголовки, body, JSON, `Content-Type` и CORS на прикладном уровне.
3. Методы `GET`, `POST`, `PUT`, `PATCH`, `DELETE`: выбор метода по действию.
4. Коды ответов `200`, `201`, `204`, `400`, `401`, `403`, `404`, `422`, `500`.
5. REST как договорённость: ресурсы, имена путей, path- и query-параметры.
6. FastAPI: приложение, роут, Uvicorn, автоматическая документация.
7. Pydantic-схемы: входные данные, выходные данные, валидация и сообщения ошибок.
8. CRUD в памяти: идентификатор, поиск, обновление, удаление, проверки границ.
9. Ошибки API: `HTTPException`, единый смысл `404` и `422`, что не отдавать клиенту.
10. Роутеры и организация FastAPI-проекта без преждевременного усложнения.
11. Swagger/OpenAPI и ручная проверка через Postman или Insomnia.
12. Тесты API через `TestClient`: успешный и ошибочный сценарий.

## Практикумы и проектные сессии

- Практикум 1: прочитать реальные HTTP-запросы в DevTools/Postman и объяснить их структуру.
- Практикум 2: разработать контракт API до написания endpoint: путь, метод, body, ответ и ошибки.
- Практикум 3: реализовать CRUD в памяти и проверить все endpoints через Swagger.
- Практикум 4: добавить 8–10 API-тестов на валидацию, несуществующий объект и успешный CRUD.
- Проектная сессия 1: `Planner API` с роутерами задач и категориями.
- Проектная сессия 2: привести ответы к одному стилю, дополнить README примерами запросов.

## Обязательный результат

`Planner API` без базы данных: CRUD для учебных задач, схемы входа/ответа, корректные статусы, документация FastAPI и тесты ключевых сценариев.

## Контрольная точка

Ученик объясняет отличие `GET` от `POST`, `PUT` от `PATCH`, body от query-параметра, выбирает подходящий статус, запускает сервер и воспроизводит запрос через Swagger или Postman.

## Дополнительно для активных

- Пагинация в памяти: `limit`, `offset`, метаданные ответа.
- Фильтрация и сортировка с валидацией допустимых полей.
- Версия API `/api/v1` и причины версионирования.
- Кастомный обработчик ошибки и единый формат ошибки.
- Написать маленького клиента на `httpx` или `requests` для своего API.
- Контрактные тесты на полную CRUD-цепочку.

## Не включать раньше времени

JWT, Redis, очереди, сложный async-код и «чистую архитектуру». Сначала нужно научиться строить и проверять ясный HTTP-контракт.

# Месяц 4. SQL, PostgreSQL и моделирование данных

## Цель

Дать реальную базу работы с данными. Ученик сначала понимает SQL и ограничения базы, а не воспринимает ORM как магию. В конце месяца API хранит данные в PostgreSQL.

## Обязательные уроки

1. Зачем нужна база данных: долговечность, параллельный доступ, целостность, отличие от JSON.
2. PostgreSQL, база, схема, таблица, строка, столбец, типы данных.
3. `CREATE TABLE`, первичный ключ, `NOT NULL`, `UNIQUE`, `DEFAULT`, `CHECK`; нормализация на практическом уровне и борьба с дублированием данных.
4. `INSERT`, `SELECT`, `UPDATE`, `DELETE` и безопасная параметризация запросов.
5. Фильтрация, сортировка, ограничение результата: `WHERE`, `ORDER BY`, `LIMIT`, `OFFSET`.
6. Агрегации: `COUNT`, `SUM`, `AVG`, `GROUP BY`, `HAVING`.
7. Связи один-ко-многим и многие-ко-многим; внешний ключ и целостность данных.
8. `JOIN`: какие строки вернут `INNER JOIN` и `LEFT JOIN`.
9. Транзакции, `COMMIT`, `ROLLBACK` и почему несколько связанных действий должны быть атомарны.
10. Индексы на базовом уровне: что ускоряют и почему индекс не нужен на каждую колонку.
11. Подключение Python/FastAPI к PostgreSQL через параметризованные запросы.
12. Отладка данных: читать SQL-ошибки, проверять данные в `psql` или GUI, делать дамп схемы.

## Практикумы и проектные сессии

- Практикум 1: создать учебную схему и написать 20 SQL-запросов вручную.
- Практикум 2: спроектировать таблицы задач, категорий и учебных целей; защитить схему перед наставником.
- Практикум 3: написать запросы с `JOIN`, агрегацией и фильтрацией на тестовых данных.
- Практикум 4: воспроизвести нарушение ограничения базы и объяснить, почему база его отклонила.
- Проектная сессия 1: перенести CRUD задач из памяти в PostgreSQL.
- Проектная сессия 2: добавить категорию, фильтрацию и один отчётный endpoint.

## Обязательный результат

`Planner API + PostgreSQL`: API сохраняет задачи и категории в базе, данные переживают перезапуск, SQL-запросы параметризованы, связи и ограничения описаны в отдельном документе.

# Месяц 6. Авторизация, права доступа и системное тестирование

4. Логин и токен: JWT как подписанное утверждение, срок жизни, ограничения JWT.

7. Роли и разрешения: минимальная модель `user` / `admin`; не строить сложную RBAC-систему раньше времени.
8. Угрозы базового уровня: утечки секретов, слабые пароли, лишние данные в ответе, доверие данным клиента.
9. Стратегия тестирования: unit, integration и API-тест; что проверять на каждом уровне.
10. `pytest`: фикстуры, тестовая база, изоляция данных, Arrange-Act-Assert.
11. Тестирование авторизации и прав: нет токена, неверный токен, чужой объект, успешный сценарий.
12. Анализ неудачного теста: минимальный воспроизводимый случай, исправление, регрессионный тест.

## Практикумы и проектные сессии

## Формат проверки работы наставником

В конце каждого месяца наставник проверяет не только то, что код запускается. Минимальный чек-лист:

- ученик может объяснить ключевое решение своими словами;
- проект запускается по инструкции с чистого состояния;
- есть негативные сценарии и обработка ожидаемых ошибок;
- код разделён по понятной ответственности, без необъяснимой «магии»;
- коммиты отражают ход работы, а README соответствует реальности;
- замечания из ревью исправлены отдельным осмысленным изменением.

Если ученик не может объяснить код, решение не считается усвоенным даже при внешне рабочем результате. В таком случае наставник возвращает не весь проект, а один конкретный пробел: например, выбор статуса HTTP, работа `JOIN`, зависимость текущего пользователя или изоляция тестовых данных.

## Рискованные места и методические решения

| Риск | Решение в плане |
| --- | --- |
| Ученик знает термины, но не пишет код | Каждая тема заканчивается артефактом, тестом или проектной частью |
| Слишком ранняя архитектура | Абстракции появляются только после работыющего простого решения |
| ORM скрывает основы SQL | Сначала месяц SQL и PostgreSQL, затем SQLAlchemy |
| Авторизация воспринимается как «вставить готовый код» | До JWT разбираются пароли, владелец ресурса, негативные сценарии и тесты |
| Docker появляется в последний день | Docker и CI вынесены в отдельный месяц после устойчивых тестов и API |
| Активные ученики скучают | Есть изолированный дополнительный блок, не ломающий основной темп |
| Слишком много новых технологий | Redis и фоновые задачи появляются только после завершённого API и базового релиза |

## Итоговая позиция методиста

Самый реалистичный маршрут для ученика с нуля при 20 часах в неделю — **девять месяцев**. Его критерий успеха не количество пройденных тем, а способность самостоятельно собрать, проверить, объяснить и передать другому человеку один аккуратный backend-проект.

Шестой месяц — хорошая промежуточная точка и возможный выход на стажировки для сильного ученика. Девятый месяц — момент, когда можно уверенно начинать массовые отклики, если проект защищён, GitHub не пустой, а ученик способен решать базовые практические задачи без подсказки.
